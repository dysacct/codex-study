学习目标

学 PostgreSQL 的关键不是背命令，而是建立一条清晰的思考链：数据如何建模，SQL 如何表达问题，数据库如何保证一致性，查询为什么快或慢，系统如何稳定运行。

新手阶段

能安装 PostgreSQL，使用 psql，创建数据库、表、约束，并完成增删改查。

进阶阶段

能写 JOIN、聚合、窗口函数、CTE，理解事务隔离级别和常见并发问题。

熟练阶段

能用索引和 EXPLAIN 分析性能，做备份恢复、权限管理、迁移和应用接入。

推荐学习顺序

  1. 用 Docker 或本机安装启动 PostgreSQL。
  2. 用一套业务小模型反复练习 CRUD 和查询。
  3. 引入约束、事务和索引，开始解释“为什么”。
  4. 使用 EXPLAIN、备份、权限、连接池处理真实项目问题。

环境安装

PostgreSQL 是一个客户端/服务端数据库。服务端负责存储和执行 SQL，客户端如 psql、GUI 工具或应用程序负责连接它。

使用 Docker 快速启动

这种方式适合学习和项目试验，能避免本机环境差异。

shell
# 启动一个 PostgreSQL 容器，密码仅用于本地学习
docker run --name pg-study \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=study \
  -p 5432:5432 \
  -d postgres:18

# 进入 psql 命令行客户端
docker exec -it pg-study psql -U postgres -d study

# 停止和删除学习容器
docker stop pg-study
docker rm pg-study

常见错误：端口 5432 被占用。可以改为 -p 5433:5432，之后连接本机 5433。

psql 必会命令
命令用途
\l列出数据库
\c study连接到指定数据库
\dt列出当前 schema 的表
\d users查看表结构
\x切换扩展显示，适合看宽表
\timing显示 SQL 执行时间
\q退出 psql

SQL 基础

SQL 是声明式语言。你描述“要什么结果”，数据库负责选择执行方式。新手最重要的是养成显式列名、明确条件、先查后改的习惯。

创建一个学习库模型

sql
-- 创建用户表：每个用户有唯一邮箱
CREATE TABLE users (
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  name TEXT NOT NULL,
  email TEXT NOT NULL UNIQUE,
  status TEXT NOT NULL DEFAULT 'active',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 创建订单表：每个订单属于一个用户
CREATE TABLE orders (
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  user_id BIGINT NOT NULL REFERENCES users(id),
  amount NUMERIC(12, 2) NOT NULL CHECK (amount >= 0),
  paid_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

增删改查

sql
-- 插入多行数据，并返回生成的 id
INSERT INTO users (name, email)
VALUES
  ('Ada', 'ada@example.com'),
  ('Linus', 'linus@example.com')
RETURNING id, name, email;

-- 查询活跃用户，永远优先写明确列名
SELECT id, name, email, created_at
FROM users
WHERE status = 'active'
ORDER BY created_at DESC;

-- 修改单个用户，WHERE 条件要足够精确
UPDATE users
SET status = 'disabled'
WHERE email = 'linus@example.com'
RETURNING id, status;

-- 删除前先 SELECT 检查命中范围
DELETE FROM users
WHERE status = 'disabled'
RETURNING id, email;

常见错误

忘记 WHERE 的 UPDATE 和 DELETE 会影响整张表。实际工作中，先用同样条件写 SELECT count(*) 看范围，再执行修改。

表设计与约束

表设计的目标是让非法数据尽早失败。应用层可以做校验，但数据库约束是最后一道一致性边界。

常用数据类型

  • BIGINT：自增主键或大整数。
  • TEXT：变长文本，业务系统中通常够用。
  • NUMERIC(p, s)：金额等精确小数。
  • TIMESTAMPTZ：带时区语义的时间点，推荐用于事件时间。
  • JSONB：半结构化数据，但不要把所有字段都塞进去。

常用约束

  • PRIMARY KEY：唯一标识一行。
  • NOT NULL：字段必须有值。
  • UNIQUE：保证业务唯一性。
  • CHECK：保证字段范围或状态集合。
  • FOREIGN KEY：保证表之间引用存在。
sql
-- 使用 CHECK 限制状态枚举，避免脏状态进入表
ALTER TABLE users
ADD CONSTRAINT users_status_check
CHECK (status IN ('active', 'disabled', 'deleted'));

-- 为订单添加业务编号，保证唯一且可读
ALTER TABLE orders
ADD COLUMN order_no TEXT UNIQUE;

-- 给外键设置级联策略要谨慎，这里选择禁止删除仍有订单的用户
ALTER TABLE orders
DROP CONSTRAINT orders_user_id_fkey,
ADD CONSTRAINT orders_user_id_fkey
FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE RESTRICT;
规范化与反规范化

规范化是把重复数据拆到独立表，靠外键引用，优点是一致性强。反规范化是为了查询性能或报表便利保留冗余字段，优点是读取简单。熟练的判断方式是：先保证模型正确，再用明确的性能证据决定是否冗余。

查询能力

查询是 PostgreSQL 的日常核心。越复杂的查询越要分层表达：先过滤，再连接，再聚合，最后排序分页。

JOIN：把表重新组合成业务视图

sql
-- 查询每个用户的订单总额，LEFT JOIN 会保留没有订单的用户
SELECT
  u.id,
  u.name,
  COALESCE(SUM(o.amount), 0) AS total_amount
FROM users AS u
LEFT JOIN orders AS o ON o.user_id = u.id
GROUP BY u.id, u.name
ORDER BY total_amount DESC;

CTE：让复杂 SQL 更可读

sql
-- 先计算付费订单，再按用户汇总
WITH paid_orders AS (
  SELECT user_id, amount
  FROM orders
  WHERE paid_at IS NOT NULL
)
SELECT
  user_id,
  COUNT(*) AS paid_order_count,
  SUM(amount) AS paid_amount
FROM paid_orders
GROUP BY user_id;

窗口函数：保留明细同时计算排名

sql
-- 给每个用户的订单按金额排名
SELECT
  id,
  user_id,
  amount,
  ROW_NUMBER() OVER (
    PARTITION BY user_id
    ORDER BY amount DESC
  ) AS amount_rank
FROM orders;
分页：OFFSET 与游标式分页

LIMIT/OFFSET 简单，但页数越深越慢，因为数据库仍然需要跳过前面的行。业务列表更推荐“游标式分页”，使用上一页最后一条记录的排序键继续查询。

sql
-- 游标式分页：稳定、可走索引、适合无限滚动
SELECT id, created_at, amount
FROM orders
WHERE created_at < TIMESTAMPTZ '2026-01-01 00:00:00+00'
ORDER BY created_at DESC, id DESC
LIMIT 20;

事务与并发

事务把多条 SQL 变成一个原子操作。它要么全部成功，要么全部失败。数据库用隔离级别和锁来处理多个会话同时读写的问题。

sql
-- 一个付款流程：订单付款和用户状态修改必须一起成功
BEGIN;

UPDATE orders
SET paid_at = now()
WHERE id = 101 AND paid_at IS NULL
RETURNING id, paid_at;

UPDATE users
SET status = 'active'
WHERE id = 5;

COMMIT;

-- 如果中间发现数据不对，可以 ROLLBACK 回滚
-- ROLLBACK;
隔离级别适合场景注意点
READ COMMITTED多数 Web 业务默认选择同一事务内两次查询可能看到不同提交结果
REPEATABLE READ需要事务内快照稳定可能出现序列化失败，需要重试策略
SERIALIZABLE强一致性、金融类关键流程并发冲突更多，应用必须能重试事务
行锁：避免同一资源被重复处理
sql
-- 多个 worker 抢任务时，跳过已被锁定的任务
BEGIN;

SELECT id, payload
FROM jobs
WHERE status = 'pending'
ORDER BY id
FOR UPDATE SKIP LOCKED
LIMIT 10;

-- 处理完任务后标记完成
UPDATE jobs
SET status = 'done'
WHERE id IN (1, 2, 3);

COMMIT;

索引与性能

索引让数据库更快定位行，但也会增加写入成本和存储空间。不要“所有字段都建索引”，要根据查询条件、排序方式和基数来设计。

B-tree：默认索引 GIN：JSONB / 数组 / 全文 BRIN：超大顺序数据 Partial：部分索引 Expression：表达式索引
sql
-- 查询经常按 user_id 和 created_at 过滤/排序时，建立复合索引
CREATE INDEX idx_orders_user_created
ON orders (user_id, created_at DESC);

-- 只索引未付款订单，减少索引体积
CREATE INDEX idx_orders_unpaid
ON orders (created_at)
WHERE paid_at IS NULL;

-- 邮箱大小写不敏感查询可以用表达式索引
CREATE INDEX idx_users_lower_email
ON users (lower(email));

用 EXPLAIN 看执行计划

sql
-- BUFFERS 能显示读取了多少共享缓冲区，有助于判断 IO 压力
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, amount, created_at
FROM orders
WHERE user_id = 5
ORDER BY created_at DESC
LIMIT 20;

调优顺序

先确认慢 SQL 和真实参数，再看执行计划，然后评估索引、统计信息、SQL 写法和数据量。不要只凭感觉建索引。

高级能力

PostgreSQL 不只是关系数据库，也提供 JSONB、全文搜索、视图、函数、触发器、分区等能力。熟练使用的标准是：知道何时用，也知道何时不用。

JSONB：存储半结构化属性
sql
-- 给用户增加 profile，用于保存可变属性
ALTER TABLE users
ADD COLUMN profile JSONB NOT NULL DEFAULT '{}';

-- 查询 JSONB 字段中的城市
SELECT id, name, profile ->> 'city' AS city
FROM users
WHERE profile @> '{"city": "Shanghai"}';

-- 为 JSONB 包含查询建立 GIN 索引
CREATE INDEX idx_users_profile_gin
ON users USING gin (profile);
视图：封装稳定查询
sql
-- 把常用报表查询封装成视图
CREATE VIEW user_order_summary AS
SELECT
  u.id AS user_id,
  u.name,
  COUNT(o.id) AS order_count,
  COALESCE(SUM(o.amount), 0) AS total_amount
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.name;

-- 像普通表一样查询视图
SELECT *
FROM user_order_summary
ORDER BY total_amount DESC;
分区表：管理大表

分区适合按时间或范围增长的超大表，例如日志、订单流水、事件数据。它主要解决维护、裁剪和归档问题，不是所有慢查询的万能药。

sql
-- 按月份创建范围分区父表
CREATE TABLE events (
  id BIGINT GENERATED ALWAYS AS IDENTITY,
  event_time TIMESTAMPTZ NOT NULL,
  payload JSONB NOT NULL
) PARTITION BY RANGE (event_time);

-- 创建 2026 年 1 月分区
CREATE TABLE events_2026_01
PARTITION OF events
FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');

运维与安全

会写 SQL 只是第一步。生产环境还要考虑备份、恢复、权限、连接数、监控和迁移。

备份与恢复

shell
# 逻辑备份单个数据库，适合迁移和中小库备份
pg_dump -h localhost -U postgres -d study -Fc -f study.dump

# 恢复到新数据库，先创建目标库
createdb -h localhost -U postgres study_restore
pg_restore -h localhost -U postgres -d study_restore study.dump

权限最小化

sql
-- 为应用创建独立用户，避免使用超级用户连接
CREATE ROLE app_user LOGIN PASSWORD 'change-me';

-- 只授予必要权限
GRANT CONNECT ON DATABASE study TO app_user;
GRANT USAGE ON SCHEMA public TO app_user;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;
生产环境检查清单
  • 应用连接使用普通账号，不使用 postgres 超级用户。
  • 定期演练恢复，备份没恢复过就不算真正可靠。
  • 打开慢查询日志，定位真实慢 SQL。
  • 控制连接数，应用侧使用连接池。
  • 迁移脚本可回放、可审查，避免手工改生产表。

应用连接

应用连接 PostgreSQL 时，重点是参数化查询、连接池、超时和事务边界。永远不要把用户输入拼成 SQL 字符串。

go
// 使用 pgxpool 连接 PostgreSQL，示例省略错误处理细节之外的业务代码
package main

import (
  "context"
  "fmt"
  "log"

  "github.com/jackc/pgx/v5/pgxpool"
)

func main() {
  // 连接串应放在环境变量中，避免写死密码
  dsn := "postgres://app_user:change-me@localhost:5432/study"
  pool, err := pgxpool.New(context.Background(), dsn)
  if err != nil {
    log.Fatal(err)
  }
  defer pool.Close()

  // 参数化查询用 $1、$2，防止 SQL 注入
  var name string
  err = pool.QueryRow(context.Background(),
    "SELECT name FROM users WHERE email = $1",
    "ada@example.com",
  ).Scan(&name)
  if err != nil {
    log.Fatal(err)
  }

  fmt.Println(name)
}
javascript
// 使用 node-postgres 的连接池执行参数化查询
import pg from "pg";

const pool = new pg.Pool({
  // 生产环境建议从环境变量读取 DATABASE_URL
  connectionString: "postgres://app_user:change-me@localhost:5432/study",
  max: 10,
  idleTimeoutMillis: 30000
});

// $1 是参数占位符，不要拼接用户输入
const result = await pool.query(
  "SELECT id, name FROM users WHERE email = $1",
  ["ada@example.com"]
);

console.log(result.rows);

练习项目

熟练来自“带约束的反复练习”。下面的项目从小到大，建议每个都写建表脚本、种子数据、查询脚本和性能分析记录。

1. 图书借阅系统

设计 books、members、loans，实现借书、还书、逾期查询。

2. 电商订单系统

设计用户、商品、订单、订单项、支付记录，练习事务和库存扣减。

3. 内容标签系统

实现文章、标签、多对多关系，练习 JOIN、聚合和全文搜索。

4. 任务队列

使用 FOR UPDATE SKIP LOCKED，实现多个 worker 并发抢任务。

5. 用户行为日志

使用分区表和 JSONB，练习大表查询、归档和统计。

6. 慢查询诊断

构造百万级订单数据，用 EXPLAIN 对比无索引、单列索引、复合索引。

综合练习 SQL

sql
-- 练习：找出最近 30 天消费最高的 10 个用户
WITH recent_paid_orders AS (
  SELECT user_id, amount
  FROM orders
  WHERE paid_at >= now() - INTERVAL '30 days'
)
SELECT
  u.id,
  u.name,
  COUNT(*) AS order_count,
  SUM(r.amount) AS total_amount
FROM recent_paid_orders r
JOIN users u ON u.id = r.user_id
GROUP BY u.id, u.name
ORDER BY total_amount DESC
LIMIT 10;

总结

从新手到熟练的路线可以浓缩为三句话：用约束保护数据，用 SQL 清晰表达问题，用执行计划验证性能判断。PostgreSQL 很强，但真正让系统可靠的是你对数据边界、事务边界和查询边界的理解。
