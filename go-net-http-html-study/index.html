Step 1 · 新概念：服务器会“听端口”

1. 最小服务器：让 Go 开门营业

What：HTTP 服务器就是一个一直等请求的程序。它监听某个端口，比如 8080。

Why：没有服务器，浏览器或 curl 就像敲一扇不存在的门，当然没人回应。

Principle：http.ListenAndServe(":8080", nil) 会启动服务。第二个参数是 handler；这里传 nil 表示使用默认路由器。

go · main.go
package main

import (
	"fmt"
	"net/http"
)

func main() {
	// 当用户访问 / 时，执行这个函数。
	http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
		// w 是响应出口，往里面写什么，客户端就收到什么。
		fmt.Fprintln(w, "hello, Go HTTP")
	})

	fmt.Println("open: http://localhost:8080")
	// :8080 表示监听本机 8080 端口。
	if err := http.ListenAndServe(":8080", nil); err != nil {
		panic(err)
	}
}

Step 2 · 新概念：handler 三件套

2. handler 三件套：w、r、函数本体

What：handler 是处理请求的函数。它通常长这样：func(w http.ResponseWriter, r *http.Request)。

Why：请求来了，总得有人看请求、算结果、写响应。handler 就是那位干活的同学。

Principle：r 是输入，包含方法、路径、Header、Body；w 是输出，用来写状态码、Header、Body。

go · main.go
package main

import (
	"fmt"
	"net/http"
)

func hello(w http.ResponseWriter, r *http.Request) {
	// r.Method 是请求方法，例如 GET。
	// r.URL.Path 是请求路径，例如 /hello。
	fmt.Fprintf(w, "method=%s path=%s\n", r.Method, r.URL.Path)
}

func main() {
	http.HandleFunc("/hello", hello)

	fmt.Println("open: http://localhost:8080/hello")
	if err := http.ListenAndServe(":8080", nil); err != nil {
		panic(err)
	}
}

Step 3 · 新概念：ServeMux 是分流前台

3. 多路由：不同路径找不同人

What：路由把不同 URL 分配给不同 handler。

Why：真实项目不可能只有一个接口。健康检查、用户、订单、登录，都得各走各的门。

Principle：http.NewServeMux() 创建一个路由器。mux.HandleFunc 注册路径和处理函数。

go · main.go
package main

import (
	"fmt"
	"net/http"
)

func main() {
	// 显式创建 mux，项目里比使用默认全局 mux 更清晰。
	mux := http.NewServeMux()

	mux.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
		fmt.Fprintln(w, "ok")
	})

	mux.HandleFunc("/users", func(w http.ResponseWriter, r *http.Request) {
		fmt.Fprintln(w, "users list")
	})

	fmt.Println("open: http://localhost:8080/health")
	if err := http.ListenAndServe(":8080", mux); err != nil {
		panic(err)
	}
}
shell · 验证
curl http://localhost:8080/health
curl http://localhost:8080/users

Step 4 · 新概念：HTTP Method 是动作

4. GET / POST：同一个资源，不同动作

What：GET 常用于查询，POST 常用于创建。

Why：路径像名词，方法像动词。/users 是“用户”，GET 是“看用户”，POST 是“创建用户”。

Principle：Go 1.22+ 的 ServeMux 支持把方法写进 pattern，例如 GET /users。

go · main.go
package main

import (
	"fmt"
	"net/http"
)

func main() {
	mux := http.NewServeMux()

	mux.HandleFunc("GET /users", func(w http.ResponseWriter, r *http.Request) {
		fmt.Fprintln(w, "list users")
	})

	mux.HandleFunc("POST /users", func(w http.ResponseWriter, r *http.Request) {
		// 创建成功通常返回 201。
		w.WriteHeader(http.StatusCreated)
		fmt.Fprintln(w, "create user")
	})

	if err := http.ListenAndServe(":8080", mux); err != nil {
		panic(err)
	}
}
shell · 验证
curl -i http://localhost:8080/users
curl -i -X POST http://localhost:8080/users

Step 5 · 新概念：Query 是问问题的小纸条

5. 查询参数：/search?q=go

What：查询参数是 URL 中 ? 后面的内容，比如 q=go。

Why：搜索、分页、排序、过滤都适合用查询参数。它们是在问：“我要哪一批数据？”

Principle：r.URL.Query().Get("q") 读取参数，不存在时返回空字符串。

go · main.go
package main

import (
	"fmt"
	"net/http"
)

func main() {
	mux := http.NewServeMux()

	mux.HandleFunc("GET /search", func(w http.ResponseWriter, r *http.Request) {
		// 读取 ?q=xxx 中的 q。
		q := r.URL.Query().Get("q")
		if q == "" {
			http.Error(w, "missing q", http.StatusBadRequest)
			return
		}

		fmt.Fprintf(w, "you searched: %s\n", q)
	})

	if err := http.ListenAndServe(":8080", mux); err != nil {
		panic(err)
	}
}
shell · 验证
curl -i "http://localhost:8080/search?q=go"
curl -i "http://localhost:8080/search"

Step 6 · 新概念：PathValue 是门牌号

6. 路径参数：/users/{id}

What：路径参数把 URL 的某一段当变量，例如 /users/42 的 42。

Why：查询具体资源时，路径参数比 query 更自然：/users/42 就像“42 号用户”。

Principle：Go 1.22+ 支持 {id} pattern，用 r.PathValue("id") 读取。

go · main.go
package main

import (
	"fmt"
	"net/http"
)

func main() {
	mux := http.NewServeMux()

	mux.HandleFunc("GET /users/{id}", func(w http.ResponseWriter, r *http.Request) {
		// id 来自路径中的 {id}。
		id := r.PathValue("id")
		fmt.Fprintf(w, "user id is %s\n", id)
	})

	if err := http.ListenAndServe(":8080", mux); err != nil {
		panic(err)
	}
}
shell · 验证
curl http://localhost:8080/users/42

Step 7 · 新概念：状态码和 Header 是说明书

7. Header 与状态码：别只会写 body

What：状态码说明结果，Header 说明响应元信息，Body 才是正文。

Why：客户端不是靠猜的。201 表示创建成功，400 表示请求有问题，Content-Type 表示 body 格式。

Principle：先设置 Header，再调用 WriteHeader，最后写 body。顺序乱了，Go 会提前发默认 200。

go · main.go
package main

import (
	"fmt"
	"net/http"
)

func main() {
	mux := http.NewServeMux()

	mux.HandleFunc("POST /users", func(w http.ResponseWriter, r *http.Request) {
		// 1. 先设置 Header。
		w.Header().Set("Content-Type", "text/plain; charset=utf-8")
		w.Header().Set("X-Lesson", "headers-before-body")

		// 2. 再设置状态码。
		w.WriteHeader(http.StatusCreated)

		// 3. 最后写响应体。
		fmt.Fprintln(w, "created")
	})

	if err := http.ListenAndServe(":8080", mux); err != nil {
		panic(err)
	}
}
shell · 验证
curl -i -X POST http://localhost:8080/users

Step 8 · 新概念：返回 JSON

8. JSON 响应：接口开始像接口了

What：JSON 是 Web API 最常用的数据格式。

Why：前端、移动端、其他后端服务都能轻松解析 JSON。纯文本适合打招呼，JSON 适合干活。

Principle：结构体字段用 json tag 控制输出字段名，json.NewEncoder(w).Encode 写 JSON。

go · main.go
package main

import (
	"encoding/json"
	"net/http"
)

type User struct {
	ID   int64  `json:"id"`
	Name string `json:"name"`
}

func main() {
	mux := http.NewServeMux()

	mux.HandleFunc("GET /users/{id}", func(w http.ResponseWriter, r *http.Request) {
		id := r.PathValue("id")
		user := User{ID: 1, Name: "user-" + id}

		// 告诉客户端：我返回的是 JSON。
		w.Header().Set("Content-Type", "application/json; charset=utf-8")
		_ = json.NewEncoder(w).Encode(user)
	})

	if err := http.ListenAndServe(":8080", mux); err != nil {
		panic(err)
	}
}
shell · 验证
curl -i http://localhost:8080/users/42

Step 9 · 新概念：读取 JSON Body

9. JSON 请求体：客户端也能递菜单

What：请求体是客户端提交的正文。创建用户时，名字就适合放在 JSON body 里。

Why：复杂数据不要塞 URL。URL 适合定位资源，Body 适合提交数据。

Principle：json.NewDecoder(r.Body).Decode(&req) 读取 JSON。读取前用 http.MaxBytesReader 限制大小，别让一个请求撑爆厨房。

go · main.go
package main

import (
	"encoding/json"
	"net/http"
)

type CreateUserRequest struct {
	Name string `json:"name"`
}

type CreateUserResponse struct {
	ID   int64  `json:"id"`
	Name string `json:"name"`
}

func main() {
	mux := http.NewServeMux()

	mux.HandleFunc("POST /users", func(w http.ResponseWriter, r *http.Request) {
		// 最多读取 1MB 请求体，防止超大 body。
		r.Body = http.MaxBytesReader(w, r.Body, 1<<20)
		defer r.Body.Close()

		var req CreateUserRequest
		if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
			http.Error(w, "invalid json", http.StatusBadRequest)
			return
		}
		if req.Name == "" {
			http.Error(w, "name is required", http.StatusBadRequest)
			return
		}

		w.Header().Set("Content-Type", "application/json; charset=utf-8")
		w.WriteHeader(http.StatusCreated)
		_ = json.NewEncoder(w).Encode(CreateUserResponse{ID: 1001, Name: req.Name})
	})

	if err := http.ListenAndServe(":8080", mux); err != nil {
		panic(err)
	}
}

Step 10 · 新概念：http.Handler 接口

10. Handler 接口：项目开始长骨架

What：http.Handler 是接口，只要求一个方法：ServeHTTP(w, r)。

Why：函数适合小 demo。项目里 handler 往往需要依赖 service、logger、db，把它们放进结构体更清晰。

Principle：结构体实现 ServeHTTP，就能被 mux.Handle 注册。

go · main.go
package main

import (
	"fmt"
	"net/http"
)

type UserHandler struct {
	// 真实项目中这里可以放 service、db、logger。
	prefix string
}

func (h *UserHandler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
	fmt.Fprintf(w, "%s users endpoint\n", h.prefix)
}

func main() {
	mux := http.NewServeMux()

	// mux.Handle 接收 http.Handler。
	mux.Handle("GET /users", &UserHandler{prefix: "demo"})

	if err := http.ListenAndServe(":8080", mux); err != nil {
		panic(err)
	}
}
shell · 验证
curl http://localhost:8080/users

Step 11 · 新概念：中间件是统一拦截器

11. 中间件：别把重复逻辑撒得到处都是

What：中间件是包装 handler 的函数，常见签名：func(http.Handler) http.Handler。

Why：日志、鉴权、限流、恢复 panic、request id 都是“每个接口都可能需要”的逻辑，应该统一处理。

Principle：在 next.ServeHTTP(w, r) 前后插入逻辑。前面像进门检查，后面像出门结账。

go · main.go
package main

import (
	"fmt"
	"log"
	"net/http"
	"time"
)

func logging(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		start := time.Now()

		// 把请求交给下一个 handler。
		next.ServeHTTP(w, r)

		// handler 结束后记录日志。
		log.Printf("%s %s cost=%s", r.Method, r.URL.Path, time.Since(start))
	})
}

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("GET /hello", func(w http.ResponseWriter, r *http.Request) {
		fmt.Fprintln(w, "hello middleware")
	})

	// logging 包住 mux，于是所有请求都会被记录。
	if err := http.ListenAndServe(":8080", logging(mux)); err != nil {
		panic(err)
	}
}

Step 12 · 新概念：Request.Context 是请求生命线

12. Context 超时：别让慢任务霸占座位

What：r.Context() 表示当前请求的生命周期。

Why：客户端断开、上游超时、服务关闭时，下游查询和耗时任务应该停下，不要继续空转。

Principle：用 context.WithTimeout 给子任务设置时间预算，耗时函数通过 select 监听 ctx.Done()。

go · main.go
package main

import (
	"context"
	"fmt"
	"net/http"
	"time"
)

func slowWork(ctx context.Context) error {
	// 这个任务本来要 2 秒。
	select {
	case <-time.After(2 * time.Second):
		return nil
	case <-ctx.Done():
		return ctx.Err()
	}
}

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("GET /slow", func(w http.ResponseWriter, r *http.Request) {
		// 只给 slowWork 500ms，超时就撤。
		ctx, cancel := context.WithTimeout(r.Context(), 500*time.Millisecond)
		defer cancel()

		if err := slowWork(ctx); err != nil {
			http.Error(w, "timeout: "+err.Error(), http.StatusGatewayTimeout)
			return
		}

		fmt.Fprintln(w, "slow work done")
	})

	if err := http.ListenAndServe(":8080", mux); err != nil {
		panic(err)
	}
}

Step 13 · 新概念：http.Server 是正式店长

13. http.Server：生产环境别裸奔

What：http.Server 是可配置的 HTTP 服务对象。

Why：http.ListenAndServe 是快捷方式，适合练习。生产环境需要超时配置，否则慢客户端可能拖住连接。

Principle：配置 ReadHeaderTimeout、ReadTimeout、WriteTimeout、IdleTimeout，给服务边界上护栏。

go · main.go
package main

import (
	"fmt"
	"net/http"
	"time"
)

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("GET /hello", func(w http.ResponseWriter, r *http.Request) {
		fmt.Fprintln(w, "configured server")
	})

	server := &http.Server{
		Addr:              ":8080",
		Handler:           mux,
		ReadHeaderTimeout: 2 * time.Second,  // 读请求头最多 2 秒。
		ReadTimeout:       5 * time.Second,  // 读完整请求最多 5 秒。
		WriteTimeout:      10 * time.Second, // 写响应最多 10 秒。
		IdleTimeout:       60 * time.Second, // keep-alive 空闲最多 60 秒。
	}

	if err := server.ListenAndServe(); err != nil {
		panic(err)
	}
}
shell · 验证
curl http://localhost:8080/hello

Step 14 · 新概念：优雅关闭

14. 优雅关闭：打烊也要等客人吃完

What：优雅关闭是在退出时停止接收新请求，并等待正在处理的请求完成。

Why：直接杀进程可能让请求处理到一半，写了一半数据，或者数据库事务还没收尾。

Principle：server.Shutdown(ctx) 关闭监听器，等待已有连接结束，直到 ctx 超时。

go · main.go
package main

import (
	"context"
	"errors"
	"fmt"
	"net/http"
	"os"
	"os/signal"
	"syscall"
	"time"
)

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("GET /hello", func(w http.ResponseWriter, r *http.Request) {
		fmt.Fprintln(w, "hello graceful shutdown")
	})

	server := &http.Server{
		Addr:              ":8080",
		Handler:           mux,
		ReadHeaderTimeout: 2 * time.Second,
	}

	go func() {
		fmt.Println("open: http://localhost:8080/hello")
		if err := server.ListenAndServe(); err != nil && !errors.Is(err, http.ErrServerClosed) {
			panic(err)
		}
	}()

	// 等 Ctrl+C 或容器终止信号。
	stop := make(chan os.Signal, 1)
	signal.Notify(stop, os.Interrupt, syscall.SIGTERM)
	<-stop

	// 最多等 5 秒让旧请求收尾。
	ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
	defer cancel()

	if err := server.Shutdown(ctx); err != nil {
		panic(err)
	}
	fmt.Println("server stopped gracefully")
}

Step 15 · 新概念：httptest

15. httptest：不用开端口也能测接口

What：net/http/httptest 是标准库测试 HTTP handler 的工具。

Why：每次手动 curl 太慢，也容易漏。自动测试能让你改代码时更放心。

Principle：httptest.NewRequest 造请求，httptest.NewRecorder 记录响应，然后直接调用 handler。

go · main.go
package main

import (
	"encoding/json"
	"net/http"
)

type HealthResponse struct {
	Status string `json:"status"`
}

func healthHandler(w http.ResponseWriter, r *http.Request) {
	// 返回一个简单 JSON，用来测试。
	w.Header().Set("Content-Type", "application/json; charset=utf-8")
	_ = json.NewEncoder(w).Encode(HealthResponse{Status: "ok"})
}

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("GET /health", healthHandler)
	_ = http.ListenAndServe(":8080", mux)
}

package main

import (

func TestHealthHandler(t *testing.T) {
	// 造一个假的 HTTP 请求。
	req := httptest.NewRequest(http.MethodGet, "/health", nil)

	// 造一个响应记录器，它会接住 handler 写出的内容。
	rec := httptest.NewRecorder()

	// 直接调用 handler，不需要真的启动 8080 端口。
	healthHandler(rec, req)

	if rec.Code != http.StatusOK {
		t.Fatalf("want 200, got %d", rec.Code)
	}

	if !strings.Contains(rec.Body.String(), `"status":"ok"`) {
		t.Fatalf("unexpected body: %s", rec.Body.String())
	}
}

常见坑：小白最容易踩的地方

先写 body 再改状态码

一旦写了 body，Go 就可能已经发送 200 OK。状态码和 Header 要先设置。

把 mux 当成可有可无

demo 可以用默认 mux，项目里建议显式创建 http.NewServeMux()，结构更干净。

读取 body 不限大小

JSON body 前加 http.MaxBytesReader。这是服务端的饭量控制，不然容易被大请求撑住。

忽略 r.Context()

数据库、RPC、慢任务都应该接收请求 context，不要让用户走了任务还在后面忙活。

生产不配 timeout

裸 ListenAndServe 适合学习。生产要用 http.Server 配超时。

只会 curl，不会测试

httptest 能让 handler 测试又快又稳。手动 curl 是验收，不是测试体系。

面试题：用人话回答，也能显得很专业

1. http.Handler 和 http.HandlerFunc 是什么关系？

答：http.Handler 是接口，要求实现 ServeHTTP。http.HandlerFunc 是函数类型，它自己实现了 ServeHTTP，所以普通函数可以被适配成 handler。简单说：接口是岗位要求，HandlerFunc 是符合要求的临时工。

2. ResponseWriter 和 Request 分别干什么？

答：*http.Request 是输入，保存方法、URL、Header、Body、Context。http.ResponseWriter 是输出，用来写状态码、Header 和响应体。

3. 为什么 Header 要在 WriteHeader 或 Write 前设置？

答：因为响应一旦开始发送，Header 和状态码就基本定了。先 Write 会隐式发送 200 OK，后面再改状态码就晚了。

4. 为什么生产环境要用 http.Server？

答：http.Server 能配置超时、Handler、优雅关闭等生产参数。裸 ListenAndServe 是快捷方式，适合学习和 demo，不适合严肃服务。

5. 中间件的本质是什么？

答：中间件是 func(http.Handler) http.Handler，它接收一个 handler，返回一个增强后的 handler。核心就是在 next.ServeHTTP 前后做通用逻辑。

6. Request.Context 有什么用？

答：它表示请求生命周期。客户端断开、请求超时、服务关闭时，context 会取消。下游数据库查询、RPC、耗时任务应该接收它，及时停止无意义工作。

总结：把 net/http 放进脑子里的正确姿势

从今天开始，看到 Go HTTP 服务，你就按这条线理解：Server 开门，ServeMux 分流，Handler 干活，Request 是输入，ResponseWriter 是输出，Context 管生命周期，httptest 保证你改代码不心慌。

最后一句口诀

路径决定找谁，方法决定干啥，Request 带来问题，ResponseWriter 交回答，Context 负责喊停，Server 负责稳住全场。
