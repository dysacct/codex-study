Go context：从入门知识点到生产级用法

`context` 是 Go 后端工程里贯穿请求生命周期的核心工具。它不负责“杀死 goroutine”，而是负责传递取消信号、截止时间和少量请求级元数据。学会它，你才能写出不会无故阻塞、不会泄漏、能被上游超时控制的服务代码。

核心心智模型

先记住一句话：context 是请求生命周期的控制信号树。父 context 被取消，所有派生出来的子 context 都会收到取消信号。

What

context.Context 是一个接口，用来跨 API 边界传递取消信号、截止时间和请求级值。

Why

后端请求经常跨 goroutine、HTTP、RPC、数据库。上游取消或超时后，下游工作应尽快停止。

Problem

没有 context 时，慢查询、阻塞网络调用、后台 goroutine 可能继续跑，造成资源泄漏和雪崩。

学习路线

  1. 先理解接口的四个方法。
  2. 再掌握取消、超时、deadline 的传播。
  3. 然后学习 Value 的边界，不把它当参数袋。
  4. 最后进入 HTTP、数据库、中间件、面试题和生产排错。

Context 接口

Go 官方标准库中，Context 的本体很小，真正的复杂性来自它如何在函数调用链中被持续传递。

What / Principle / Common Mistakes
方法含义生产用法
Deadline()返回截止时间和是否存在截止时间判断请求剩余时间，避免启动注定超时的工作
Done()返回一个只读 channel，取消时关闭在 select 中监听取消
Err()返回取消原因：Canceled 或 DeadlineExceeded区分用户取消和系统超时
Value(key)读取请求级键值传 trace id、request id、认证主体等

常见错误：只创建 context，却不在耗时函数内部监听 Done()。context 不会魔法般终止业务逻辑，代码必须主动检查。

go
package main

import (
	"context"
	"fmt"
	"time"
)

func main() {
	// 创建一个 50ms 后自动取消的 context。
	ctx, cancel := context.WithTimeout(context.Background(), 50*time.Millisecond)
	defer cancel() // 释放 timer 资源，是生产代码必备习惯。

	// Done 返回一个 channel，取消或超时时会被关闭。
	select {
	case <-ctx.Done():
		fmt.Println("stopped:", ctx.Err())
	case <-time.After(200 * time.Millisecond):
		fmt.Println("work completed")
	}
}

Background 与 TODO

context.Background() 和 context.TODO() 都返回不可取消、无截止时间、无值的根 context，但语义不同。

Background

用于 main 函数、初始化、测试、顶层请求入口。它表示“这里就是上下文树的根”。

TODO

用于暂时不知道应该传什么 context 的地方。它是一种标记：未来应该补上真正的调用链 context。

go
package main

import (
	"context"
	"fmt"
)

func loadConfig(ctx context.Context) error {
	// 这里暂时没有耗时操作，只演示函数签名的规范。
	fmt.Println("config loaded, ctx is nil?", ctx == nil)
	return nil
}

func main() {
	// 顶层入口使用 Background。
	_ = loadConfig(context.Background())

	// 迁移旧代码时可以临时用 TODO，但不要长期保留。
	_ = loadConfig(context.TODO())
}

WithCancel：主动取消

WithCancel 返回一个子 context 和一个 cancel 函数。调用 cancel() 后，子 context 和它的所有后代都会收到取消信号。

What / Why / Problem / Principle

What：创建一个可以被手动取消的 context。

Why：调用方提前拿到结果、用户断开连接、任务失败时，要通知下游停止工作。

Problem：如果 worker 只管循环发送数据，消费者退出后 worker 会阻塞并泄漏。

Principle：cancel() 会关闭 Done() channel，监听该 channel 的 goroutine 可以自然退出。

go
package main

import (
	"context"
	"fmt"
)

func numbers(ctx context.Context) <-chan int {
	out := make(chan int)
	go func() {
		defer close(out) // goroutine 退出时关闭结果 channel。
		for i := 1; ; i++ {
			select {
			case <-ctx.Done():
				return // 收到取消信号，避免 goroutine 泄漏。
			case out <- i:
			}
		}
	}()
	return out
}

func main() {
	ctx, cancel := context.WithCancel(context.Background())
	defer cancel()

	for n := range numbers(ctx) {
		fmt.Println(n)
		if n == 3 {
			cancel() // 主动通知生产者停止。
		}
	}
}

生产提醒

只要你调用了 WithCancel、WithTimeout 或 WithDeadline，通常都应该 defer cancel()。这不仅是取消任务，也会释放父 context 对子 context 的引用以及相关 timer。

WithTimeout / WithDeadline：时间边界

超时是服务稳定性的边界。没有超时的服务调用，在故障时会把 goroutine、连接池和请求队列慢慢拖满。

WithTimeout：相对时间 WithDeadline：绝对时间点 Err：DeadlineExceeded
go
package main

import (
	"context"
	"fmt"
	"time"
)

func slowQuery(ctx context.Context) error {
	// 模拟一个可能很慢的外部调用。
	select {
	case <-time.After(200 * time.Millisecond):
		fmt.Println("query success")
		return nil
	case <-ctx.Done():
		return ctx.Err() // 把超时原因返回给调用方。
	}
}

func main() {
	// 给下游调用 80ms 的预算。
	ctx, cancel := context.WithTimeout(context.Background(), 80*time.Millisecond)
	defer cancel()

	if err := slowQuery(ctx); err != nil {
		fmt.Println("query failed:", err)
	}
}
Real World Usage
  • HTTP handler 从 r.Context() 继承客户端连接生命周期。
  • 调用数据库时传入 QueryContext，让慢查询能被取消。
  • 调用第三方 API 时给每次请求设置独立超时。
  • 批处理任务给每个子任务设置预算，避免单个任务拖垮整体。

WithValue：请求级值

WithValue 很有用，也很容易被滥用。它应该只保存跨 API 和进程边界的请求级元数据，例如 request id、trace id、认证用户，而不是普通函数参数。

最重要的边界

不要用 context 传业务参数，例如分页大小、商品 ID、开关配置。普通参数要写在函数签名里，让依赖显式可见。

go
package main

import (
	"context"
	"fmt"
)

// 使用私有 key 类型，避免和其他包的 key 冲突。
type requestIDKey struct{}

func withRequestID(ctx context.Context, id string) context.Context {
	// Value 适合保存请求级元数据。
	return context.WithValue(ctx, requestIDKey{}, id)
}

func requestIDFrom(ctx context.Context) (string, bool) {
	id, ok := ctx.Value(requestIDKey{}).(string)
	return id, ok
}

func main() {
	ctx := withRequestID(context.Background(), "req-20260604-001")
	id, ok := requestIDFrom(ctx)
	fmt.Println(id, ok)
}
适合放入 context不适合放入 context
request id、trace id、span id函数可直接接收的业务参数
认证主体、租户 id可选参数、配置项、分页条件
日志字段、链路追踪字段数据库连接、全局客户端、缓存对象

Cancel Cause：带原因的取消

Go 1.20 引入了 WithCancelCause 和 Cause。它们解决的问题是：只看 ctx.Err() 通常只能知道“取消了”或“超时了”，却不知道业务上为什么取消。

go
package main

import (
	"context"
	"errors"
	"fmt"
)

var ErrQuotaExceeded = errors.New("quota exceeded")

func main() {
	ctx, cancel := context.WithCancelCause(context.Background())

	// 业务层可以记录更具体的取消原因。
	cancel(ErrQuotaExceeded)

	<-ctx.Done()
	fmt.Println("err:", ctx.Err())
	fmt.Println("cause:", context.Cause(ctx))
}
Interview Questions

问：ctx.Err() 和 context.Cause(ctx) 有什么区别？

答：Err() 返回标准取消状态，通常是 context.Canceled 或 context.DeadlineExceeded。Cause 可以返回通过 CancelCauseFunc 设置的业务错误；如果没有设置 cause，通常退化为和 Err() 一样的结果。

AfterFunc：取消后的回调

context.AfterFunc(ctx, f) 会在 context 被取消后，用新的 goroutine 执行回调函数。它适合把“取消信号”桥接到一些不直接支持 context 的阻塞 API。

go
package main

import (
	"context"
	"fmt"
	"time"
)

func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 30*time.Millisecond)
	defer cancel()

	done := make(chan struct{})
	stop := context.AfterFunc(ctx, func() {
		// context 取消后，这个函数会在独立 goroutine 中执行。
		fmt.Println("cleanup after cancel:", ctx.Err())
		close(done)
	})
	defer stop() // 如果 ctx 还没取消，可以停止关联。

	<-done
}

Common Mistakes

stop() 不会等待已经开始执行的回调结束。如果回调需要完成确认，要自己用 channel、WaitGroup 或其他同步方式协调。

WithoutCancel：断开取消传播

context.WithoutCancel(parent) 创建一个继承 Value、但不受父 context 取消影响的 context。它适合请求结束后还要做短小、可控的后台清理，例如异步记录审计日志。

go
package main

import (
	"context"
	"fmt"
)

type traceKey struct{}

func main() {
	parent := context.WithValue(context.Background(), traceKey{}, "trace-001")
	ctx, cancel := context.WithCancel(parent)
	cancel()

	// detached 仍能读取 Value，但不会因为 parent cancel 而 Done。
	detached := context.WithoutCancel(ctx)
	fmt.Println("trace:", detached.Value(traceKey{}))
	fmt.Println("err:", detached.Err())
}

生产边界

不要用 WithoutCancel 绕过正常取消，让后台任务无限跑。真正的后台任务仍然应该有自己的超时、队列、重试和关闭机制。

HTTP 与数据库实战

生产项目里，context 最常见的入口是 HTTP 请求。客户端断开连接、网关超时、handler 返回，都会通过 r.Context() 影响下游调用。

go
package main

import (
	"context"
	"database/sql"
	"encoding/json"
	"net/http"
	"time"

	_ "github.com/lib/pq"
)

type User struct {
	ID   int64  `json:"id"`
	Name string `json:"name"`
}

func userHandler(db *sql.DB) http.HandlerFunc {
	return func(w http.ResponseWriter, r *http.Request) {
		// 从请求继承 context，再给数据库查询单独设置预算。
		ctx, cancel := context.WithTimeout(r.Context(), 200*time.Millisecond)
		defer cancel()

		var u User
		err := db.QueryRowContext(ctx,
			"SELECT id, name FROM users WHERE id = $1",
			1,
		).Scan(&u.ID, &u.Name)
		if err != nil {
			// 如果是超时或客户端取消，可以按网关策略返回。
			http.Error(w, err.Error(), http.StatusGatewayTimeout)
			return
		}

		_ = json.NewEncoder(w).Encode(u)
	}
}

func main() {
	// 示例代码需要本地 PostgreSQL；重点是 QueryRowContext 的使用方式。
	db, _ := sql.Open("postgres", "postgres://user:pass@localhost/db?sslmode=disable")
	defer db.Close()
	http.HandleFunc("/users/1", userHandler(db))
	_ = http.ListenAndServe(":8080", nil)
}

中间件与链路追踪

中间件适合把 request id、trace id、认证主体放进 context。下游日志、数据库访问、RPC 调用都能从同一个 context 中取到链路信息。

go
package main

import (
	"context"
	"fmt"
	"net/http"
	"time"
)

type requestIDKey struct{}

func requestIDMiddleware(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		// 实际项目应使用 UUID 或网关传入的 request id。
		rid := fmt.Sprintf("req-%d", time.Now().UnixNano())
		ctx := context.WithValue(r.Context(), requestIDKey{}, rid)
		next.ServeHTTP(w, r.WithContext(ctx))
	})
}

func handler(w http.ResponseWriter, r *http.Request) {
	// handler 内部读取请求级元数据。
	rid, _ := r.Context().Value(requestIDKey{}).(string)
	_, _ = w.Write([]byte("request id: " + rid))
}

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("/", handler)
	_ = http.ListenAndServe(":8080", requestIDMiddleware(mux))
}

常见错误

把 context 存进 struct

context 应该作为函数第一个参数显式传递，通常命名为 ctx。存进 struct 会模糊生命周期。

传 nil context

不要传 nil。暂时不知道传什么时，使用 context.TODO()。

忘记 cancel

派生了可取消 context 后忘记调用 cancel，可能造成 timer、子节点引用和 goroutine 生命周期问题。

用 Value 传参数

Value 不是可选参数容器。业务参数应该显式出现在函数签名里。

只传不监听

耗时循环、阻塞发送、外部调用要主动监听 ctx.Done() 或使用支持 context 的 API。

后台任务继承请求 context

请求结束后，后台任务会跟着取消。需要异步任务时，要设计独立生命周期和超时。

Practice：练习题

1. 可取消生成器

写一个不断生成订单号的 goroutine，消费者收到 5 个后取消 context，确认 goroutine 退出。

2. HTTP 超时

写一个 handler，内部模拟 300ms 任务，然后给它 100ms context timeout，观察返回错误。

3. 数据库预算

使用 QueryContext 包装一次查询，给查询独立 200ms 预算，并记录超时日志。

4. Trace ID 中间件

写一个中间件生成 request id，放入 context，并在 handler 和日志函数里读取。

5. Cause 排错

用 WithCancelCause 在业务校验失败时记录具体错误，并在上层打印 context.Cause。

6. AfterFunc 清理

模拟一个阻塞资源，在 context 取消后执行清理函数，并用 channel 等待清理完成。

Interview Questions：面试题

1. context 的核心作用是什么？

它用于在 API 边界和 goroutine 之间传递取消信号、截止时间和请求级值。核心价值是让上游能够控制下游工作生命周期，避免泄漏和无意义的继续执行。

2. context 会杀死 goroutine 吗？

不会。context 只是关闭 Done() channel 并暴露错误状态。goroutine 必须主动在 select 中监听 ctx.Done() 并返回。

3. 为什么 context 通常作为第一个参数？

这是 Go 生态的约定，能让函数生命周期依赖显式可见，也方便静态分析工具检查 context 是否正确传递。典型签名是 func Do(ctx context.Context, arg Arg) error。

4. WithTimeout 后为什么还要 defer cancel？

因为 cancel 不只表示“取消”，还会释放 context 相关资源，例如 timer 和父子引用。即使超时最终会发生，也应该在操作完成时尽早释放。

5. WithValue 的最佳实践是什么？

只传请求级元数据；key 使用自定义私有类型；读取时做类型断言检查；不要传业务参数、配置、数据库连接或全局客户端。

6. 父 context 取消后，子 context 会怎样？

所有从父 context 派生出来的子 context 都会被取消。取消是单向向下传播，子 context 取消不会反向取消父 context。

总结

Go context 的生产级理解可以压缩成五条：显式传递、及时 cancel、耗时操作监听 Done、Value 只放请求级元数据、超时是服务稳定性的边界。

最终口诀

ctx 不是参数袋，不会杀 goroutine；它是取消信号树，是请求生命周期的预算表。
